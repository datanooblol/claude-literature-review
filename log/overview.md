# Papers Overview

| Title | Problem | Methodology | Conclusion |
|-------|---------|-------------|------------|
| [[Golden-Retriever High-Fidelity Agentic Retrieval Augmented Generation]] | Standard RAG misinterprets domain-specific jargon and abbreviations in user queries, causing poor document retrieval that post-retrieval fixes cannot correct. | Reflection-based question augmentation: identify jargon via LLM, look up a jargon dictionary, identify query context via CoT, rewrite the augmented question before retrieval; paired with offline LLM-driven document summarization. | Pre-retrieval question augmentation yields 57.3% avg improvement over Vanilla LLM and 35.0% over Vanilla RAG across three LLMs on a domain-specific Q&A dataset. |
| [[Parametric Retrieval Augmented Generation]] | All RAG methods inject knowledge via input context, but LLMs store knowledge in parameters — causing inefficiency and a fundamental gap in how well external knowledge can be utilized. | Offline document parameterization via LoRA adapters trained on augmented (rewritten + QA-pair) documents; online Retrieve-Update-Generate pipeline merges adapter weights into FFN before generation. | P-RAG outperforms in-context RAG baselines on most benchmarks with 29–36% lower inference latency; combining parametric and in-context injection achieves highest overall performance. |
| [[Let's (not) just put things in Context Test-Time Training for Long-Context LLMs]] | Long-context LLMs fail to retrieve relevant information buried in long inputs, and inference-time "thinking" tokens show rapid diminishing returns as context grows. | Formalize score dilution in static self-attention; propose query-only test-time training (qTTT) that caches K/V once and updates only query projection matrices via few gradient steps on short sampled spans. | qTTT consistently outperforms FLOP-matched thinking tokens on LongBench-v2 and ZeroScrolls, with up to 14.1% average gain for Qwen3-4B; retrieval-driven tasks benefit most. |
| [[Oreo A Plug-in Context Reconstructor to Enhance Retrieval-Augmented Generation (2025)]] | Retrieved chunks contain noise, redundancy, and scattered evidence that misleads the generator, while the retriever and generator remain unaligned due to independent optimization. | Plug-in T5-small module trained via SFT → contrastive multi-task learning → PPO RL alignment that reconstructs retrieved chunks into concise, query-specific context before generation. | Oreo outperforms all compression/extraction baselines across 5 QA datasets, improving EM by up to 37.5% on NQ while compressing context 84–94% and reducing latency 23–43%. |
| [[Context Length Alone Hurts LLM Performance Despite Perfect Retrieval]] | LLM long-context failures are assumed to stem from retrieval failures, but the role of input length itself — independent of retrieval quality — is unexplored. | Controlled synthetic experiments with essay, whitespace, and masked distractors across 5 models and 4 tasks; retrieval verified by exact match; mitigation via retrieve-then-solve (recite evidence then use short context). | Even with perfect retrieval and no distraction (masked attention), accuracy drops 13.9–85% as context grows — the sheer length alone degrades performance; reciting evidence before solving recovers up to 31.2%. |
| [[Needle Threading Can LLMs Follow Threads through Near-Million-Scale Haystacks]] | Current long-context benchmarks saturate too quickly, cover only sub-100k contexts, and lack granular insights — leaving the true limits of frontier LLMs unknown. | Synthetic UUID key-value haystack tasks of increasing difficulty (single needle → multiple → conditional → threading → multi-threading) on 17 LLMs at up to 630k tokens; introduces effective context length metric in characters. | Most models' effective context length is far below their advertised limit; LLMs are surprisingly thread-safe (tracking multiple simultaneous chains does not hurt per-thread performance); forward threads are easier than backward; tokenizer differences make cross-model token comparisons misleading. |
| [[REFRAG Rethinking RAG based Decoding]] | RAG inference suffers from high TTFT latency because retrieved passages dominate context length, but RAG's block-diagonal sparse attention structure is not exploited by generic long-context methods. | Compress retrieved chunks into pre-computable encoder embeddings fed to the decoder instead of tokens; three-stage training (reconstruction → CPT with curriculum → RL-based selective expansion); no decoder architecture changes needed. | 30.85× TTFT acceleration over LLaMA and 3.75× over previous SOTA (CEPE) at k=32 compression without perplexity loss; extends LLaMA-2-7B context from 4k to 16k+ and improves RAG task performance at equal latency. |
| [[Sufficient Context A New Lens on Retrieval Augmented Generation Systems]] | RAG errors are attributed to retrieval or generation failures, but the concept of "sufficient context" (whether retrieved content actually contains enough to answer the query) has never been formally defined or measured. | Define sufficient context formally; build a Gemini 1.5 Pro autorater (93% accuracy); stratify FreshQA, Musique, HotpotQA responses into Correct/Abstain/Hallucinate by sufficiency; propose selective generation combining sufficient context + model confidence. | Even with sufficient context, LLMs hallucinate at 14–17% rates; with insufficient context, models hallucinate far more than abstain; 35–62% of insufficient-context answers are still correct; selective generation improves accuracy 2–10% at most coverage levels. |
| [[On the Influence of Context Size and Model Choice in Retrieval-Augmented Generation Systems]] | There is no systematic study of how context size, LLM choice, and retrieval method jointly affect RAG performance on long-form QA tasks requiring synthesis of multiple evidence snippets. | Vary gold snippets (0–10), closed-retrieval (8k-doc corpus), and open retrieval (10M PubMed + Wikipedia) for BM25 vs. semantic search, across 8 LLMs on BioASQ and QuoteSum long-form QA benchmarks. | Performance improves up to ~15 snippets then saturates; biomedical tasks favor Mixtral/Mistral/Qwen over GPT-4o regardless of model size; encyclopedic tasks favor GPT-4o/LLaMA; BM25 beats semantic search for precision-critical domains; open retrieval remains far below gold-snippet performance. |
| [[Reconstructing Context Evaluating Advanced Chunking Strategies for Retrieval-Augmented Generation]] | Fixed-size chunking before embedding fragments semantic context; late chunking and contextual retrieval both aim to preserve global context but their comparative trade-offs in full RAG pipelines are unstudied. | Compare early vs. late chunking (RQ1) and traditional vs. contextual retrieval with rank fusion + reranking (RQ2) on NFCorpus and MsMarco with four embedding models. | Contextual retrieval with rank fusion outperforms late chunking on retrieval metrics but requires high computational cost (LLM per chunk, 20GB VRAM); late chunking is efficient but not consistently better than early chunking across all models. |
| [[Enhancing Retrieval-Augmented Generation A Study of Best Practices]] | Best practices for RAG system design across factors like LLM size, prompt, chunk size, KB size, retrieval stride, query expansion, and in-context learning are not well understood. | 74-run ablation study across 9 design dimensions on TruthfulQA and MMLU using Mistral 7B/45B; propose novel Contrastive ICL RAG and Focus Mode (sentence-level retrieval). | Contrastive In-Context Learning RAG (correct + incorrect examples in context) is the best performer across all metrics; Focus Mode (sentence-level retrieval) ranks second; chunk size, KB size, retrieval stride, and query expansion have minimal impact. |
